Никогда не фильтровать данные после получения из БД. Если данные попали в API-ответ — они уже доступны через curl/Postman. RLS = WHERE clause в SQL.

4 · Timezone Strategy

  • Все timestamps в БД хранятся в UTC
  • Каждый бренд имеет поле reporting_timezone
  • Группировки применяют AT TIME ZONE brand.reporting_timezone
  • Неделя начинается в Понедельник (ISO 8601)
  • UI отображает время в UTC + tooltip с local timezone

5 · Currency Conversion

Rate lock в момент транзакции. Не пересчитывать исторические данные при изменении курса.

Обязательные поля для каждой транзакции:

original_amount DECIMAL(18,4) original_currency VARCHAR(3) -- ISO 4217 eur_amount DECIMAL(18,4) eur_rate DECIMAL(18,6) rate_date DATE

7 · Incremental ETL & Late Data

  • Today's data — инкрементальные обновления каждые 1–5 мин
  • Завершённые периоды — полный пересчёт ночным job (не инкрементально)
  • Late-arriving data — cashouts, affiliate batch, бонусные корректировки: обрабатываются отдельным reconciliation job

8 · Audit Trail

Write-only лог. Нельзя изменить или удалить запись. Retention: 5 лет.

Что логировать: просмотр отчётов · экспорт · изменение фильтров · вход в систему · изменение прав · доступ к финансовым данным

9 · URL State Management

  • Все фильтры, группировки, drill-down уровни сериализуются в query params
  • history.pushState() без перезагрузки страницы
  • Читаемый формат — не base64 blob
  • При шаринге ссылки автоматически применяются ограничения прав получателя
Example URL

/reports/affiliate?brand=brandx&period=2026-03&group=media_item&geo=DE,AT&sort=ngr_desc

11 · API Design Contract

Стиль: REST, ориентированный на вычисления (не CRUD). Каждый endpoint = один отчёт.

URL versioning: /api/v1/... с первого дня. Breaking changes → /api/v2/. Старый endpoint живёт минимум 3 месяца с header Sunset: <date>.

Response Envelope — единый формат

// SUCCESS { "data": { ... }, "meta": { "generated_at": "2026-03-12T14:30:00Z", "data_freshness": "2026-03-12T14:28:55Z", "cache_hit": true, "request_id": "req_abc123" }, "pagination": { "cursor": "eyJpZCI6MTIzfQ==", "has_more": true, "total": 6957 } } // ERROR { "error": { "code": "FILTER_INVALID", "message": "Brand 'test' not found or access denied", "request_id": "req_abc123" } }
⚠ Pagination

Только cursor-based pagination (after_cursor). Offset-based ломается при real-time данных. Максимум 500 строк на страницу. Больше → async export.

12 · DB Schema — Partitioning & Indexes

Решение, которое нельзя изменить без миграции после запуска:

Партиционирование

Рекомендация: PARTITION BY (brand_id, created_at::month) для таблиц transactions, deposits, cashouts. Каждый запрос читает один бренд за один месяц — не всю таблицу.

Обязательные индексы (создать вместе со схемой)

-- Core transaction queries idx_tx_brand_date ON transactions (brand_id, created_at) idx_tx_player ON transactions (player_id, created_at) idx_tx_affiliate ON transactions (affiliate_id, created_at) idx_tx_status_type ON transactions (status, type, created_at) -- Cohort analysis idx_ftd_cohort ON players (brand_id, ftd_date) -- Affiliate report idx_aff_mi_geo ON aff_performance (affiliate_id, media_item_id, geo, period)
🚫 Важно

Индексы создаются вместе со схемой, а не после наполнения данными. Добавление индексов на заполненную таблицу = блокировка на часы.

13 · ETL Idempotency

Если ETL запустился дважды (сбой, retry) — данные не должны дублироваться.

  • Каждая запись в raw-слое хранит source_id — уникальный ID в источнике
  • Только UPSERT ON CONFLICT (source_id) DO UPDATE — никогда просто INSERT
  • Pre-aggregation jobs: сначала DELETE, потом INSERT (не прибавлять к старому)
  • ETL job логирует: job_id, started_at, finished_at, status, rows_processed, error

Ежедневная проверка дублей: GROUP BY source_id HAVING COUNT(*) > 1 → alert в Slack если найдены.

14 · Data Freshness Indicator

Пользователь всегда должен знать, насколько свежие данные на экране.

СостояниеИндикатор в UITrigger
Свежие (<5 мин)Зелёная точка + «Data as of 14:28»—
Устаревшие (5–15 мин)Жёлтая точка + «Updating...»—
Неактуальные (>15 мин)Красный баннер + «Data may be stale»Alert в Slack

API: каждый ответ содержит meta.data_freshness. Frontend считает разницу с сервером, не с browser clock.

Health endpoint: GET /api/v1/health → последняя успешная синхронизация по каждому бренду.

17 · Alert Architecture

Цепочка: Data Source → Metric Evaluator (job each 5 min) → Alert Rule Engine → Notification Dispatcher → Email / Telegram / In-app

ТребованиеОписание
ДедупликацияЕсли AR% ниже порога 30 минут подряд — один alert, не шесть
CooldownМинимум 30 мин до следующего (configurable per rule)
ИсторияТаблица alert_history: кто получил, когда, значение метрики в момент триггера
AcknowledgeIn-app alert можно отметить как прочитанный
МаршрутизацияПо отделу, бренду, роли

Alert Rule Structure

metric: "ar_percent" scope: {brand: "brandx", child_system: "*"} condition: "< 60" window: "15min" channels: ["telegram", "in-app"] recipients: ["role:payment_team"] cooldown_minutes: 30

19 · Feature Flags

Новые отчёты нельзя выкатывать сразу всем. Feature flags решают это без деплоя.

  • Новый отчёт добавляется в код за feature_flag: "affiliate_report"
  • Super Admin включает flag в UI-панели без деплоя
  • Отчёт виден только выбранным пользователям/брендам
  • После полного rollout — flag удаляется из кода

Table: feature_flags

flag_key VARCHAR PRIMARY KEY -- 'affiliate_report', 'churn_v2' enabled BOOLEAN DEFAULT FALSE enabled_for JSONB -- {"brand_ids": [1,2], "user_ids": [99]} rollout_pct INT DEFAULT 0 -- % пользователей created_at TIMESTAMPTZ updated_by INT -- user_id

21 · MVP Definition

БлокСодержимое
Инфра (must-have)Auth (SSO), Row-Level Security, API v1, ETL pipeline, Кэш (Redis), Audit log
Отчёт #1 (MVP)Daily Performance Report — самодостаточный, все 49 метрик
Отчёт #2 (MVP)In/Out Report — платёжная команда нуждается в первую очередь
Post-MVPPayment Report, Churn Cohort, Affiliate, Alert System
✓ MVP ≠ упрощённая архитектура

MVP — самый узкий scope при полной (правильной) архитектуре. Кэш, RLS и audit log — обязательны даже для первого релиза.

22 · Load Expectations

Без этих цифр невозможно подобрать размер инфраструктуры.

ПараметрОжидаемое значение
Активных пользователей в час пика~50–100 (уточнить)
Пик одновременных запросов~200 req/sec
Объём таблицы transactions~10M строк/год
Рост данных~3× в год
DB Choice

При ~10M строк/год + 3× рост — обычный PostgreSQL достаточен на 1–2 года. Если цифры выше — ClickHouse или TimescaleDB обязательны с первого дня.
